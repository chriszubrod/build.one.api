## Architecture Decisions (April 2026)

- **Multi-repo structure.** The project is split into three repos under `/Users/chris/Applications/build.one/`: `build.one.api` (this repo), `build.one.web` (React frontend), `build.one.ios` (iOS app). Each is an independent GitHub repo.
- **API response envelope.** All API endpoints return standardized envelopes: `{"data": [...], "count": N}` for lists, `{"data": {...}}` for single entities, `{"status": "accepted", ...}` for 202. Use helpers from `shared/api/responses.py`.
- **Lookup endpoint.** `GET /api/v1/lookups?include=vendors,projects,...` returns slim dropdown data. Used by React frontend. See `shared/api/lookups.py`.
- **Jinja is gone.** As of 2026-04-24 (Wave E5) all Jinja web controllers, the `templates/` tree, `static/` directory, `get_current_user_web`, `require_module_web`, `WebAuthenticationRequired`, `RefreshRequired`, and the `Jinja2` dependency have been removed. React (`build.one.web`) is the only UI surface. Do not reintroduce Jinja templates or `require_module_web` — use `require_module_api` on every router.
- **React frontend.** `build.one.web` uses React + Vite + TypeScript. Dev server talks directly to the prod API via `VITE_API_BASE_URL`. All UI ships through React — the Jinja migration completed 2026-04-24.
- **Old AI layer was removed; new `intelligence/` layer replaces it.** The legacy AI surface is gone (`shared/ai/`, `integrations/azure/ai/`, `entities/qa`, `entities/search`, `entities/anomaly`, `entities/categorization`, `ClaudeExtractionService`, the project-resolution agent, and the attachment `ExtractionService`). Azure OpenAI, Document Intelligence, AI Search, and embeddings are all gone. A new `intelligence/` package has been built from scratch (see section below). **Do not reintroduce LangChain/LangGraph** — the new layer uses HTTPX directly against provider APIs.
- **Email-agent pipeline (2026-04-26; reshaped 2026-05-06 — three-signal classification + non-blocking draft create).** Polled invoice inbox → email_specialist agent (3 signals: email metadata + sender history + DI) → bill_specialist delegation → draft Bill in ~30–60 s end-to-end. The Function App's `poll_email_inbox` (5m cadence) calls `POST /api/v1/admin/email/poll` which lists messages tagged `Blue category` in the shared `invoice_inbox_email`, persists EmailMessage + EmailAttachment rows, and uploads attachments to blob. Separately `process_email_inbox` (1m cadence, inner-drain) calls `POST /api/v1/admin/email/process_one` which claims the oldest pending email and kicks off an email_specialist agent run.
  - **DI: prebuilt-layout with keyValuePairs (NOT prebuilt-invoice).** Generic-shape extractor returns `{content, key_value_pairs, tables, pages_count, raw}` for any document type — invoice / credit memo / statement / receipt / non-financial. `integrations/azure/document_intelligence/business/service.py::_hoist_and_validate` is a thin pass-through; doc-type classification + field extraction are the agent's job. Idempotent — re-running on an already-extracted attachment short-circuits to the cached result (saves ~6 s + the DI charge).
  - **email_specialist tools** (6 total): `read_email_message`, `search_email_sender_history(from_email, exclude_public_id?)` ← prior-context lookup, `extract_email_attachment` ← DI on demand, `record_extracted_fields(attachment_id, vendor_name?, invoice_number?, …)` ← persist agent's typed-field interpretation onto `EmailAttachment.Di*` columns, `bridge_email_attachment` ← EmailAttachment → Attachment via hash-dedup'd `EmailAttachmentBridgeService` (no blob copy; bridge layer normalizes `application/octet-stream` + `.pdf` filename → `application/pdf` so downstream PDF-only validators don't reject), `mark_email_outcome(public_id, outcome, classification, decided_action, classification_reason, confidence)` ← stamps the agent's *semantic* decision + workflow status + Outlook category. Plus the single delegation tool `delegate_to_buildone_orchestrator` (2026-06-25): every entity action is wrapped in an `EntityActionEnvelope` and handed to Build.One, which routes by `entity_type` — the legacy per-specialist `delegate_to_*_specialist` tools were removed.
  - **Controlled vocabularies** persisted on `EmailMessage` so future emails from the same sender are informed by past decisions:
    - `AgentClassification` ∈ {`vendor_invoice` | `vendor_credit_memo` | `vendor_statement` | `vendor_expense_receipt` | `customer_payment` | `customer_question` | `customer_dispute` | `internal_reply` | `internal_forward` | `vendor_newsletter` | `contract_labor_timesheet` | `non_actionable` | `unknown`}
    - `AgentDecidedAction` ∈ {`delegated_to_bill_specialist` | `delegated_to_bill_credit_specialist` | `delegated_to_expense_specialist` | `delegated_to_contract_labor_specialist` | `flagged_needs_review` | `marked_irrelevant` | `marked_processed`}
    - `AgentClassificationConfidence` DECIMAL(5,4) — agent's overall confidence; ≥0.95 routes per classification, else `needs_review` regardless of value
    - `AgentClassificationReason` NVARCHAR(1024) — one-sentence why
  - `EmailMessage.AgentSessionId` is now properly linked back at `mark_email_outcome` time (the tool forwards `ctx.session_id` to the API — was always NULL before 2026-05-06).
  - **`ReadEmailSenderHistory` sproc** powers the sender-keyed prior-context tool. Returns aggregate counts by ProcessingStatus / AgentClassification / AgentDecidedAction + counts of committed Bills/Expenses/BillCredits sourced from prior emails by this sender + distinct Vendor rows transitively associated via committed Bills. Eliminates the cold-start problem for first-of-its-kind senders.
  - **Outcome categories** applied back to Outlook via `mail_client.set_categories` (append, don't replace — preserves human's input `Blue category`). Multi-attachment outcome rollup precedence: `awaiting_approval > needs_review > processed > irrelevant`.
  - The legacy `entities/inbox`, `entities/email_thread`, `entities/classification_override`, `entities/review_entry`, the old `email_intake` workflow, and the old `templates/inbox/` + `templates/agent/` Jinja trees are still gone — the new pipeline is `entities/email_message/` (schema + repos + tools) and `intelligence/agents/email_specialist/` (agent + prompt). DI lives in `integrations/azure/document_intelligence/`. `integrations/ms/mail` is actively used.
  - **EmailMessage identity (Migration 004+005, 2026-05-27)**: stable primary key is `(InternetMessageId, Folder)`, **not** `GraphMessageId`. MS Graph recycles `GraphMessageId` values in shared mailboxes (delete-restore cycles); keying `UpsertEmailMessage`'s MERGE on it caused 160 wrong-vendor rows in May 2026 by overwriting unrelated emails when ids were reused. `InternetMessageId` is the RFC 5322 Message-ID header — globally unique + immutable across all SMTP servers. `UpsertEmailMessage` now RAISERRORs on missing IMID; the polling pipeline's `_ingest_messages` catches that and logs the email as an error rather than inserting under the broken key. `GraphMessageId` is a **mutable secondary** field — updated on every successful match. Lookup index `IX_EmailMessage_GraphMessageId` (non-unique) preserves `ReadEmailMessageByGraphMessageId` performance. Filtered unique index `UQ_EmailMessage_InternetMessageId_Folder` (`WHERE InternetMessageId IS NOT NULL`) enforces the new identity at the DB layer. See TODO.md P0 entry under "Email-agent + bill_specialist follow-ups" + `scripts/reconcile_email_message_graph_id_drift.py` for the audit script.
- **bill_specialist non-blocking-draft pattern (2026-05-06).** `create_bill` and `add_bill_line_items` are NO LONGER approval-gated. Drafts have no external side effects (no QBO, no SharePoint, no Excel — only `complete_bill` triggers those, and `complete_bill` IS still approval-gated). This unblocks the email-driven flow which previously hung 5+ min on every approval timeout.
  - **`Vendor.Notes` and `Project.Notes` (NVARCHAR(MAX))** — free-text per-entity notes, editable in the React Vendor/Project edit pages and surfaced to the bill_specialist / project_specialist agents via `FindVendorForInvoice` / `FindProjectForInvoice`. Use for vendor-specific rules (e.g. "trim trailing /N suffix from invoice numbers" for Walker Lumber) or project-specific guidance (e.g. address aliases — "also referred to as 'Bluebird Landing'"). The agent applies vendor notes first, then project notes, both verbatim. (Renamed from `Vendor.IntakeNotes` 2026-05-07; old name is gone.)
  - **`FindVendorForInvoice` sproc** + tool — multi-strategy ranked Vendor lookup (`domain_contact`@1.00 → `exact_name`@0.95 → `exact_abbreviation`@0.90 → `prefix_name`@0.85 → `substring_two_words`@0.75 → `substring_first_word`@0.65). Returns up to 5 ranked candidates with `notes` per row. **Use this — not `search_vendors` — for invoice-driven vendor binding.**
  - **`FindProjectForInvoice` sproc** + tool (on project_specialist) — ranked Project lookup matching Ship To addresses against `Project.Name` (which encodes the address — e.g. `"TB3 - 917 Tyne Blvd"`). bill_specialist reaches it via `delegate_to_project_specialist`.
  - **Inline summary-line fields on `create_bill` API** — `line_description` / `line_quantity` / `line_rate` / `line_amount` / `line_markup` / `line_price` / `line_is_billable` / `line_sub_cost_code_id` / `line_project_public_id`. Server populates the auto-created `BillLineItem` (the one carrying the attachment via `BillLineItemAttachment`) with these values directly. For invoice flows: `qty=1`, `rate=total`, `amount=qty*rate`, `markup=null`, `price=amount`, `is_billable=true`, `description` is a 6-word category summary (e.g. `"Lumber, subfloor, hangers, anchors, delivery"`) folding all DI line items. The full PDF is the per-line detail; reviewers read it for breakdowns.
  - **`Bill.Memo` template (invoice flows)**: minimal — only fields not captured in typed columns. Format: `"DOC#:{raw_invoice_number} | Ref:{po_or_job_or_reference}"`. Skip either field when not applicable. Vendor / BillNumber / Total / Project / IntakeSource live in their typed columns. Match-strategy / confidence audit data lives in the `AgentSession` transcript, not the memo.
  - **Duplicate-Bill source-email backfill** (`LinkBillSourceEmailMessage` sproc + `BillRepository.link_source_email_message`). When `BillService.create()` hits the (vendor, bill_number) uniqueness conflict and the caller passed a `source_email_message_public_id`, the existing Bill's `SourceEmailMessageId` gets stamped (only when currently NULL — won't overwrite a different link). Preserves the email-trail audit when the same Bill came in via two paths (e.g., bill_folder pipeline + email pipeline).
  - **Reviewer-reply automation (Wave 3, 2026-05-07).** When a Project Manager / Owner replies to a forwarded review notification, email_specialist detects the reply (internal-domain sender + `Re:` subject + tracked-conversation match via `find_bill_by_conversation_id`), parses approval/rejection intent + SubCostCode shorthand + description from the new-text portion of the body, and delegates to bill_specialist. Bill_specialist resolves the SCC via `find_sub_cost_code_for_reply` (segment-padded number / alias / name strategies) and calls `apply_reviewer_decision` which: validates draft state, authorizes the sender against the bill's recipient set (PM/Owner via UserProject + Role), updates the summary BillLineItem on approval (sub_cost_code + description), and transitions the Review state with `raw_reply_text` persisted in `Review.Comments`. Multi-reviewer semantics: agent applies every authorized reply (latest wins) until the human presses Complete; final review at completion catches errors. Authorization failures / final-state Reviews / non-draft Bills all classify as `internal_reply` + `marked_irrelevant`. Successful applications classify as `reviewer_reply` + `applied_reviewer_decision`. New sprocs: `ReadBillByConversationId`, `FindSubCostCodeForReply`. **NEVER assign `cost_code_id` directly** — every BillLineItem hangs off a SubCostCode; the parent CostCode is reachable via the SubCostCode.
- **`Bill.attachment_public_id` is now a universal create requirement.** Every `POST /api/v1/create/bill` (manual, agent, script) must pass an `attachment_public_id` referencing an existing Attachment row carrying the source PDF. Server creates a BillLineItem and links the attachment to it via `BillLineItemAttachment`. When `line_*` summary fields are also passed (per the inline-summary pattern above), the auto-created line is the summary line; otherwise it's an empty placeholder for the human to fill in via the Edit page. Manual UI uploads via `/upload/attachment` first to obtain the public_id; email_specialist obtains it via `bridge_email_attachment`. **Auto-bridge fallback (2026-05-26):** if the supplied `attachment_public_id` resolves to an `EmailAttachment` instead of an `Attachment` (common agent miscall — different keyspaces but identical UUID shape), `BillService.create` now invokes `EmailAttachmentBridgeService.bridge()` automatically and proceeds with the bridged Attachment. Hash-dedup ensures repeated calls don't create duplicate Attachment rows. Calling `bridge_email_attachment` explicitly is still the canonical path for the agent; the fallback is a safety net, not a license to skip it. **Other entities (Expense, BillCredit) have NOT adopted this rule yet** — only Bill.
- **`SourceEmailMessageId BIGINT NULL FK` on Bill / Expense / BillCredit.** Many-to-one back to EmailMessage; only `Bill.CreateBill` sproc currently writes it. Thread navigation via `EmailMessage.ConversationId` JOIN — no many-to-many table.
- **`core/notifications/` is gone, but `entities/device_token/` is back (2026-04-28, v0.1.0 iOS beta).** SLA scheduler stays removed. Push *registration* is reintroduced as a thin entity package: two endpoints under `/api/v1/mobile/device-token/{register,deactivate}` (auth-only, no module RBAC) backed by `dbo.DeviceToken`. **No sender yet** — tokens accumulate but nothing actually pushes through APNs. The sender lands when v0.1.x picks a trigger surface (review submitted, etc.). Do not reference `core.notifications` or `push_service`.
- **Bill folder processing — queue + tick + scheduled enumerator (2026-04-26).** The legacy `bill_agent` folder scanner is gone, but the React Bills "Process Folder" button is alive. POST inserts one `BillFolderRunItem` per PDF (`status='queued'`); the scheduler's `process_bill_folder` timer drains one item per tick (~30s cadence, inner-loop until empty or 4 min); the scheduler's `enumerate_bill_folder` timer (every 5 min) calls `/api/v1/admin/bill-folder/enumerate` so files dropped into `/source` get auto-picked-up without anyone clicking the button. Hour-window dedup on item_ids prevents stuck files from churning a fresh queue row every tick. **Expense folder processing was rebuilt 2026-06-18** as a faithful mirror of the bill-folder pipeline (`dbo.ExpenseFolderRun`/`ExpenseFolderRunItem` + `ExpenseFolderProcessor` + `/api/v1/admin/expense-folder/{tick,enumerate}` + scheduler `process_expense_folder` (30s) / `enumerate_expense_folder` (5m) timers); it no-ops harmlessly until the company's expense `/source`+`/processed` SharePoint folders are linked via the `DriveItemExpenseFolder` mapping.
- **`ProcessEngine` is CRUD-only.** Only the instant-workflow path (`execute_synchronous` / `_handle_instant_workflow`) is wired up. The `executor` property still raises `RuntimeError`; the email-intake / expense-intake / approval / timeout handlers are gone. `route()` will return `"Unknown workflow type"` for anything that isn't an instant workflow.

## Intelligence Layer (April 2026)

- **`intelligence/` package** houses all AI disciplines — agents today, room for RAG, extraction, classification, etc. Built from scratch to replace the removed AI layer. No LangChain/LangGraph; HTTPX directly against provider APIs.
- **Agent fleet (2026-04-26).** Eleven agents live in prod:
  - **buildone** (Build.One) — central orchestrator. Eleven delegation tools: `delegate_to_{sub_cost_code, cost_code, customer, project, vendor, bill, bill_credit, expense, invoice, contract_labor, time_tracking}`. No direct entity tools, no module permissions (role `Agent Orchestrator`, zero grants — makes no direct HTTP calls). Runs in two modes: **chat** (natural-language, route by word choice) and **routing** (consumes an `EntityActionEnvelope` from a trigger source and dispatches by `entity_type`). Other agents reach it via `delegate_to_buildone_orchestrator` (registered in buildone's own package, project_specialist-style).
  - **sub_cost_code_specialist** — SubCostCode CRUD + CostCode read.
  - **cost_code_specialist** — CostCode CRUD + SubCostCode read.
  - **customer_specialist** — Customer CRUD + Project read (for "what projects does X have?" child queries).
  - **project_specialist** — Project CRUD + Customer read (for parent resolution).
  - **vendor_specialist** — Vendor CRUD; **search-only reads** (`list_vendors` deliberately not exposed; ~1,100 rows would dominate context). Soft-deletes via `SoftDeleteVendorByPublicId` sproc.
  - **bill_specialist** — Bill (vendor invoice) parent CRUD + line-item CRUD + `complete_bill` workflow (locks IsDraft=false then enqueues SharePoint + Excel + QBO push via the outbox). Search-first (~18K rows). **`create_bill` and `add_bill_line_items` are NOT approval-gated** (drafts only; no external side effects). `complete_bill`, `update_bill`, `delete_bill`, `update_bill_line_item`, `remove_bill_line_item` ARE gated. For invoice-driven creation: use `find_vendor_for_invoice` (not `search_vendors`) + `delegate_to_project_specialist` + `create_bill` with inline `line_*` summary fields → single-call draft + populated summary line + attachment linked.
  - **bill_credit_specialist** — BillCredit (vendor credit memo) parent CRUD + `complete_bill_credit` workflow. Search-first (~400 rows). Same parent (Vendor), different fields (`credit_date` / `credit_number`, no due_date / no payment_term).
  - **expense_specialist** — Expense (vendor expense or refund) parent CRUD + `complete_expense` workflow. **`Expense.IsCredit=true` doubles as the "ExpenseRefund" concept** — no separate table; same specialist handles both via a single tool set. Search-first (~10K rows).
  - **invoice_specialist** — Invoice (OUR customer invoice, billed against a Project — NOT a vendor invoice) parent CRUD + `complete_invoice` workflow (PDF packet + SharePoint upload + Box mirrors + source-line billed-status sync; QBO push disabled). Search-first (~900 rows). Project read tools for parent resolution. **Unified prompt (2026-07-03):** its system prompt is the canonical `entities/invoice/intelligence/prompt.md` (Part 1 = agent surface, Part 2 = interactive InvoiceAgent playbook) — `definition.py` loads it from there; deliberately NO `prompt.md` in the agent package.
  - **email_specialist** — system-triggered orchestrator for the polled invoice inbox. **Not invoked by Build.One**; the scheduler-driven `/admin/email/process_one` kicks off one run per pending EmailMessage. Reshaped 2026-05-06 to a three-signal model (email + sender history + DI). Tools: `read_email_message`, `search_email_sender_history`, `extract_email_attachment`, `record_extracted_fields`, `bridge_email_attachment`, `mark_email_outcome`, `find_bill_by_conversation_id`, `find_contract_labor_by_conversation_id`, plus the single delegation tool `delegate_to_buildone_orchestrator`. Carries grants ONLY on Email Messages module (read+update). As of 2026-06-25 it builds an `EntityActionEnvelope` for Bills (vendor invoices), Expenses (receipts), and ContractLabor (timesheets + reviewer-replies) and hands it to Build.One; credit memos / statements still get controlled-vocabulary classifications + `flagged_needs_review` (email doesn't build BillCredit envelopes yet even though Build.One can route them).
  - Each specialist has its own narrow `Role` row with grants only on the entities it owns. Build.One's role has zero grants — it makes no direct HTTP calls.
  - Build.One's prompt routes by **literal word choice first** ("cost code" → CostCode, "sub-cost-code" → SubCostCode, "customer/client" → Customer, "project" → Project, "vendor/supplier" → Vendor, "bill"/"vendor invoice" → Bill, "bill credit"/"credit memo" → BillCredit, "expense"/"refund"/"receipt" → Expense, plain "invoice"/"customer invoice" → Invoice). Then by format/context when ambiguous.
  - **Search-vs-list discipline:** when a list response would exceed ~10K tokens / 500-1000 rows, drop list and force search. Today: Vendor + all four transactional specialists are search-only.
- **Adding another specialist** is mechanical: new `intelligence/agents/{name}_specialist/` package (definition + prompt + `__init__`), new credentials_key + DB user/role/grants (template: `seed.intelligence_agents.sql`), register `delegate_to_{name}` in buildone's `__init__.py` + append to `buildone.tools` tuple. Per-entity tools still land in `entities/{name}/intelligence/tools.py`. **Exception:** invoice_specialist has no package-local prompt — it loads the canonical `entities/invoice/intelligence/prompt.md` (shared with the interactive InvoiceAgent playbook; one prompt, one location).
- **Delegation primitive.** `intelligence/composition/delegation.py::make_delegation_tool(name, target_agent, description)` returns a Tool whose handler spawns a sub-session via `background.start_run()` with `ParentSessionId` set, subscribes to the sub's channel, forwards live events onto the parent's channel for the UI, and returns the sub's final text as the tool result. Sub-agent approvals carry their own `session_public_id` in the event so the UI POSTs decisions to the sub-session's `/approve` URL.
- **EntityActionEnvelope + Build.One routing (2026-06-25).** Trigger sources (email intake today; scheduler / MCP later) hand a structured **EntityActionEnvelope** to Build.One via `delegate_to_buildone_orchestrator` instead of calling each specialist directly — collapsing per-trigger delegate-tool sprawl into one tool. The envelope is JSON carried inside the delegation `task` string (the primitive only takes `task: str`): `{entity_type, action, intake_source, intake_source_detail, vendor/customer/project_candidate_public_id?, attachment_public_id?, payload:{task_markdown}, classification_reason, confidence}`. **Routing is by `entity_type`** (`Bill`→bill, `Expense`→expense, `BillCredit`→bill_credit, `Invoice`→invoice, `ContractLabor`/`ContractLaborLineItem`→contract_labor, `TimeEntry`/`TimeLog`→time_tracking; `ExpenseRefund`/`BillPayment` not-yet-routable). `payload.task_markdown` is the existing specialist-ready markdown the trigger source already composes, so **specialists are unchanged** — Build.One passes it through (option (a) of the bridging decision). **Failure-isolation contract:** Build.One is a pure router — it never touches entities or stamps the source record. It returns a status line (`ROUTED ok` / `ROUTED error` / `ROUTING not_routable` / `ROUTING parse_error`); the trigger source reads it and owns its own record's outcome. Schema + worked examples live in `intelligence/agents/buildone/prompt.md` ("Routing mode" section).
- **Agent-as-user auth.** Each agent is a first-class `User` + `Auth` row. At run start the orchestrator logs the agent in via `POST /api/v1/mobile/auth/login` and obtains a JWT. Every HTTP tool call routes through `shared.rbac.require_module_api()` exactly like a human request — free RBAC audit trail. Tools never call services directly; `ToolContext.call_api()` is the one and only path.
- **Least-privilege role per agent.** Build.One makes no HTTP calls, so its role has zero module grants. Each specialist's role has grants only on the entities it owns. Do not reuse Admin.
- **Tools are colocated with entities.** `entities/{name}/intelligence/tools.py` alongside the entity's `api/` / `business/` / `persistence/` / `sql/`. Self-register on import.
- **Approval UX is generic.** Any tool with `requires_approval=True` pauses the loop, emits an `ApprovalRequest` event (carrying `session_public_id`), awaits `POST /api/v1/agents/runs/{id}/approve` with `{decision: approve|reject|edit, edited_input?}`, then resumes. Each approval-gated tool supplies an `approval_summary` callable to label the card; the UI renders a form built from the tool's JSON Schema.
- **Parallel tool dispatch.** When the LLM emits multiple `tool_use` blocks in one turn, the runner runs them concurrently via `asyncio.create_task`. Events (ApprovalRequest, ApprovalDecision, ToolCallEnd) interleave live on a shared queue; `result_blocks` land in original `pending_calls` order so the LLM's tool_result history is preserved. Forwarded sub-agent events from concurrent delegations will visually interleave on Build.One's stream — UI grouping into per-source-session lanes is a separate piece of work.
- **Cross-worker safety.** Prod runs `-w 2` gunicorn; both the approval coordinator and the SSE channel registry are process-local dicts.
  - Approval coordinator: in-memory future races a DB poller on `AgentApprovalRequest.Status`; whichever signals first wins (same worker ~ms, cross-worker ~1.5s).
  - SSE `/events` falls back to `intelligence.api.replay::tail_session()` (DB-polled synthesis) when the local channel registry misses. Emits terminal `Done` only when session status is terminal; 10-min safety cap on stuck sessions.
  - `background.start_run` pre-registers the channel before returning `public_id` (via `session_channel.get_or_register`) so delegation handlers don't race the lazy channel-creation path.
  - SSE stream emits `: keepalive` comments every 15s to survive Azure proxy idle timeout during long approval waits.
- **Transport retry.** `intelligence/transport/anthropic.py` retries 429/503/529 with exponential backoff + jitter (1s/2s/4s, ±25%, Retry-After honored, 3 attempts total). 529 "Overloaded" is Anthropic's capacity signal; Sonnet hits these intermittently during high-load windows.
- **Agent credentials pattern.** Each `Agent` declares a `credentials_key` (one of `buildone_agent`, `sub_cost_code_agent`, `cost_code_agent`, `customer_agent`, `project_agent`, `vendor_agent` today); the auth helper reads `{key}_username` / `{key}_password` off `config.Settings`. Prod Azure App Service Application Settings must carry all six pairs plus `ANTHROPIC_API_KEY` and `INTERNAL_API_BASE_URL`.
- **Conversation threading.** `AgentSession.PreviousSessionId` links continuation sessions. `POST /api/v1/agents/runs/{public_id}/continue` with a follow-up `user_message` creates a new session pointing at the prior head; `intelligence/persistence/history.py::load_chain_history()` walks the chain and synthesizes canonical Messages as prior context.
- **HTTP surface** (`intelligence/api/router.py`): `POST /api/v1/agents/{name}/runs`, `GET /api/v1/agents/runs/{public_id}/events` (SSE; live from in-memory channel or DB tail), `POST /api/v1/agents/runs/{public_id}/cancel`, `POST /api/v1/agents/runs/{public_id}/continue`, `POST /api/v1/agents/runs/{public_id}/approve`. All gated on `Modules.DASHBOARD`. Cancel / continue / approve require caller to match the session's `RequestingUserId`.
- **React integration.** `build.one.web/src/agents/` holds the hook + SSE client; `BuildOneTray` renders as a right-side push drawer toggled by a header button. Conversation threading (user bubbles + agent blocks + "New" reset) + approval cards + record-block rendering are wired. `ApprovalCard.tsx` auto-builds a form from the tool's JSON Schema.
- **Schema files.** `intelligence/persistence/sql/dbo.agent_session.sql`, `dbo.agent_turn.sql`, `dbo.agent_tool_call.sql`, `dbo.agent_approval_request.sql`. All idempotent (`IF NOT EXISTS` guards + `ALTER TABLE ADD` for additive changes). View + MERGE pattern. Seed scripts: `seed.intelligence_agents.sql` (user/auth/role/grants for all three agents), `seed.buildone_role.sql` (least-privilege cutover).
- **Deploy path is Docker, not ZIP.** Prod image: `buildone-esgaducjg4d3eucf.azurecr.io/buildone:latest`. Kudu deployment history shows stale Feb 2026 entries — ignore it. Build + deploy: `az acr build --registry buildone --image buildone:latest --image buildone:<shortsha> --file Dockerfile .` then `az webapp restart --name buildone --resource-group buildone_group`. ACR handles the amd64 build server-side (no Docker daemon needed on ARM Mac). ~90s end-to-end. **Container tag must be `:latest`** — if `DOCKER_CUSTOM_IMAGE_NAME` is pinned to a specific commit tag (e.g. `:0d63b19`), `az webapp restart` silently relaunches the pinned image and `:latest` pushes are ignored. Verify with `az webapp config container show --name buildone --resource-group buildone_group --query "[?name=='DOCKER_CUSTOM_IMAGE_NAME'].value" -o tsv` — output should end with `:latest`. If pinned, repoint via `az webapp config container set ... --container-image-name buildone-esgaducjg4d3eucf.azurecr.io/buildone:latest` + restart.
