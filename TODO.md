## Cross-repo: Idempotent CREATE pattern via client UUID (v0.2, spans API + iOS)

Filed 2026-05-20 after the iOS-sync duplicate-TimeLog incident on 2026-05-19 was patched in iOS build 4 (commit pending). Build 4 closed the immediate field-worker pain via per-entity `hasAttemptedCreate` flags + natural-key pre-check on TimeEntry/TimeLog, plus the DB `UX_TimeLog_TimeEntry_ClockIn` unique index. That works but is fragile and per-entity bespoke. The proper fix is a universal client-UUID idempotency pattern.

**Problem.** Today an offline-or-poor-coverage iOS client that POSTs a CREATE can lose the response (network drop, app backgrounded mid-await, OS killing a background task). The local CD row still carries `hasPendingCreate=true`, so the next drain retries — and if the original request actually succeeded server-side, the retry creates a duplicate row. Real incident: 3 field workers got duplicate TimeLog rows on 2026-05-19. Cleanup required manual SQL across 5 users.

**Build 4 mitigation (shipped, narrow).** `CDTimeLog.hasAttemptedCreate` + `CDTimeEntry.hasAttemptedCreate` boolean flags persist across app launches. On retry, iOS pre-queries the server's existing rows for the entry/date and claims a matching one (by `(timeEntryId, clockIn, logType)` for TimeLog, `(userId, workDate)` for TimeEntry) instead of re-POSTing. Plus duplicate-key error recovery in the catch block. Plus the DB unique index on `(TimeEntryId, ClockIn)` as defense-in-depth.

**Limits of the build-4 approach:**
- **Byte-exact natural-key matching is fragile.** Any clockIn timestamp serialization drift (timezone `Z` vs `+00:00`, milliseconds present/absent) → pre-check misses → re-POST → unique-index rejection → claim path runs, but it's a brittle round-trip.
- **Only TimeLog + TimeEntry are protected.** Bill, BillLineItem, Project, Role, Module, Company, Organization, Contact, BillLineItemAttachment all have the same structural race — lower volume but same bug shape. iOS will eventually need to grow more sync surface area for field workers.
- **Per-entity matching logic.** Every new entity needs its own natural-key shape in iOS + pre-check fetch wired in.

**Proposed deeper fix — client-UUID idempotency:**

1. **API: Add `ClientId UNIQUEIDENTIFIER NOT NULL` column to every entity with user-driven CREATE.** Lightweight migration: `ALTER TABLE … ADD ClientId UNIQUEIDENTIFIER NOT NULL DEFAULT NEWID()` then a one-time backfill, then drop the default once iOS is sending it. Tables: TimeEntry, TimeLog, TimeEntryStatus (the manual-submit case), Bill, BillLineItem, BillCredit, BillCreditLineItem, Expense, ExpenseLineItem, Invoice, InvoiceLineItem, ContractLabor, ContractLaborLineItem, Project, Customer, Vendor, Contact, Role, RoleModule, Module, UserModule, UserProject, UserRole, Organization, Company, OrganizationCompany, UserOrganization, UserCompany, Attachment, BillLineItemAttachment.

2. **API: Update every CREATE sproc to be MERGE-shaped on ClientId.**
   ```sql
   IF EXISTS (SELECT 1 FROM dbo.X WHERE ClientId = @ClientId)
       SELECT existing row by ClientId
   ELSE
       INSERT … VALUES (@ClientId, …);
       SELECT inserted row;
   ```
   Plus unique index per table on ClientId. Retried CREATEs with the same ClientId are deterministic no-ops.

3. **API: Accept `client_id: UUID` on every CREATE endpoint body.** Pydantic schema; pass through to service → repo → sproc.

4. **iOS: Send `cd.publicId` (already a generated UUID) as `client_id` in every CREATE payload.** The local publicId becomes the canonical idempotency key. The server's response still returns its own `public_id` for downstream FK relationships — iOS just upserts as today.

5. **iOS: Drop the `hasAttemptedCreate` flags + per-entity natural-key matching once #4 is in.** Simpler, no bespoke code per entity.

**Cost estimate.** ~1 day API (column + sproc + endpoint payload threading on ~25 tables, mostly mechanical), ~half day iOS (CREATE payload threading + drop the bespoke pre-check logic), ~half day testing + migration. Worth doing in one focused project, not piecemeal — partial coverage doesn't help because the bug is per-entity.

**Recommended timing.** Before second customer onboards, or before substantially expanding iOS field-worker mutations (e.g., letting them create Expenses from the field, edit Projects, etc.). Not blocking, not urgent post-build-4. Note: keep the build-4 narrow protections in place until this lands — they're cheap and provide belt + suspenders.

## iOS service architecture review (v0.2.0+)

- [ ] **CoreData reads should happen on background contexts, not the main actor.** Paired with the lazy-cache refactor below. Today every service's cache-populate happens on `@MainActor` because `AppEnvironment` is `@MainActor @Observable` and every service init runs there. Cumulative 30 services × 30ms fetch = ~900ms of main-thread blocking before LoginView renders. Refactor: use `NSManagedObjectContext.perform { ... }` on a background context for the fetch, hop back to main actor with the results for UI binding. Do this at the same time as the lazy-cache refactor below — they touch the same code.
- [ ] **Move cache pre-populate out of service `init` to first-access.** Today every service in iOS `AppEnvironment.init` does a synchronous `CDX.fetchAll(in: coreData.viewContext)` to pre-populate its in-memory state. With 16 services on `@MainActor` this is fine (~100-400ms cold launch), but the cost is linear in service count + cache size. At v0.2 with 30+ services and bigger caches, this crosses the user-perceptible 1s threshold and starts blocking the LoginView from appearing. **Fix**: convert each service's init-time `fetchAll` into a `private func ensureCachePopulated()` called lazily at the top of any `loadX()` / computed-property read. Init becomes ~zero cost; cache pre-populate happens when the relevant view actually appears. Field workers who never touch Bills/Expenses/Invoices stop paying for those caches every cold launch. ~30 min per service when already touching the file. Zero behavior change for end users — they only pay for caches they navigate to.
- [ ] **Don't introduce a DI framework yet.** For 30 services, plain init injection in `AppEnvironment` is still cleaner than Swinject/Factory overhead. Revisit at ~100 services or when cold-launch construction exceeds 2s after the lazy-cache fix above.
- [ ] **Defer feature-container split until forced.** Splitting `AppEnvironment` into `CoreEnvironment` (always) + per-tab `FeatureContainer` (instantiated on tab open) is the right shape if memory ever becomes a constraint or service count reaches ~50. Until then, the eager-init pattern is right — predictable, view bodies don't nil-check, closure capture works at init time. Don't refactor pre-emptively.
- [ ] **Service init failure recovery (CoreData corruption).** Today if `CoreDataStack.init` fails to load the persistent store — corrupted file, schema-migration failure, disk error, post-iCloud-restore mismatch — the iOS app crashes inside `AppEnvironment.init` before LoginView renders. User has no in-app recovery path; only fix is delete + reinstall, which **destroys every unsynced offline edit**. For a field-worker app that's offline-first, that's the worst possible outcome. **Fix**: wrap `loadPersistentStores` in a recovery path — on failure, log the error (via a crash reporter, see next item), delete the corrupted store file, recreate fresh. User loses local cache for that session but the app launches. Pair with telemetry so we know when it's happening in the wild. Consider an "in-memory CoreData fallback" so writes stay usable for the current session even if persistence is broken. **Add a crash reporter** (Sentry / Firebase Crashlytics / a simple `POST /api/v1/admin/crash-log` endpoint) — today we have no visibility into launch crashes on testers' phones.
- [ ] **Company-switch reset must sync offline edits first.** `AppEnvironment.companyService.onCompanySwitch` (lines 174-198) calls `resetForLogout()` on every service — clears in-memory state but doesn't sync pending edits before context flips. User-initiated logout (`authService.onUserLogout`) correctly uses `prepareForLogout()` which syncs first; company switch should be at least as careful. **Scenario at Phase 5+** when CompanyId scoping is API-enforced: user in Company A clocks in offline → switches to Company B (JWT cid=B) → network returns → `observeConnectivity` fires `syncPendingUpdates` → pending clock-in POSTs with cid=B → API either rejects (edit stuck forever in CoreData) or accepts and writes the row into the wrong tenant (silent corruption). Neither outcome is good. **Fix two layers**: (1) call `prepareForLogout()` not `resetForLogout()` on company switch — same chain `onUserLogout` uses, syncs first if connected; (2) tag every pending CoreData row with a `pendingCompanyId: Int64` set at the moment of the edit, so the sync path can defer pending rows until the right Company is active again. Lands alongside Phase 5 sproc-level CompanyId enforcement on the API. Today: not a bug — only one Company exists. The moment a second Company arrives, becomes a real risk.

    **ROOT CAUSE (Phase 1 investigation, 2026-05-27):** MS Graph **reuses `GraphMessageId` values in the `invoice@rogersbuild.com` shared mailbox** when emails are deleted-then-restored (soft-delete + later undelete, or retention-policy churn). Our `UpsertEmailMessage` MERGE is keyed only on `GraphMessageId` (`MERGE ... That creates the `CreatedDatetime < ReceivedDatetime` time-inversion. The EmailAttachment rows attached to the row's previous identity stay attached, becoming phantoms tied to the now-overwritten metadata.

    **Why this affects `invoice@rogersbuild.com` specifically**: shared mailbox with aggressive cleanup (Sweep rules, retention policies, manual triage). Each delete-restore cycle frees the GraphMessageId for reuse. A personal mailbox without that churn wouldn't hit this rate of recycling.

    **Phase 3 fix (proposed):**
      1. **Switch `UpsertEmailMessage` MERGE key from `GraphMessageId` to `InternetMessageId`** (RFC 5322 globally-unique, immutable). Add unique index on `EmailMessage.InternetMessageId`.
      2. Treat `GraphMessageId` as a *mutable secondary field* that can change when Graph recycles ids. Update it on every successful poll-match.
      3. Defense-in-depth: detect identity drift — if upsert receives `(InternetMessageId=A, GraphMessageId=X)` but DB has `(InternetMessageId=B, GraphMessageId=X)`, log + INSERT a new row instead of updating. This is the path for legitimate ID recycling.
      4. Mirror the same logic on `UpsertEmailAttachment`: dedup keyed on `(parent InternetMessageId, file content hash)` rather than `(EmailMessageId, GraphAttachmentId)` — keeps EA linkage content-stable across parent overwrites.

## Bill list page review — `build.one.web/src/pages/bills/BillList.tsx` (2026-05-06)

Findings from a focused review of the Bills list page. Listed in priority order; first three are real bugs, rest are UX / hardening.

- [ ] **Vendor / Project names render blank past the dropdown's first page.** [BillList.tsx:134-135](../build.one.web/src/pages/bills/BillList.tsx#L134-L135) calls `useIdNameMap` against `/api/v1/get/vendors` and `/api/v1/get/projects` with no `?page_size=`; [useIdNameMap.ts:14](../build.one.web/src/hooks/useIdNameMap.ts#L14) just `getList`s and builds the map from one page. Pairs naturally with #1 if the list response gains `vendor_name` / `project_name`.

- [ ] **Search input has no debounce.** [BillList.tsx:386-388](../build.one.web/src/pages/bills/BillList.tsx#L386-L388) calls `setSearch` on every keystroke. `usePaginatedList` re-fetches on each change — typing "vendor" fires 6 requests against the prod API. - [ ] **Disallowed file types are dropped silently in the drop zone.** [BillList.tsx:170](../build.one.web/src/pages/bills/BillList.tsx#L170) filters by `ACCEPTED_TYPES` and skips anything else with no feedback. ## Mobile auth — refresh token UX (2026-06-02)

Surfaced during the iOS BuildOneApp.swift failure-mode review pass. Two related but distinct issues with the mobile auth flow; neither was blocking the iOS work but both are worth fixing before more field workers come online.

- [ ] **Bump `refresh_token_expire_seconds` from 86400 (24h) to ~30-90 days for the mobile flow.** Current value forces field workers to re-log in roughly every shift — 24h is a "kiosk / high-security web app" TTL, not a "field iPhone" TTL. Industry norm for mobile refresh tokens is 30-90 days because mobile devices spend long stretches in airplane mode / dead zones / closed. Biometric (Face ID) softens the re-login UX into a single tap, but only for users who opted in during onboarding AND whose biometric Keychain item hasn't been invalidated (Face ID re-enrollment invalidates it). One-line change in App Service Application Settings: `refresh_token_expire_seconds=2592000` (30d) or `7776000` (90d). The iOS code already handles whatever expiry the server returns and stores the rotated refresh_token on each `/mobile/auth/refresh` response — no client-side change needed.
  - **Web auth path is unaffected**: web uses the cookie-based refresh flow on a different endpoint (`/auth/refresh`), and the cookie's `max_age` is set from the same `refresh_token_expire_seconds`. If you want a different TTL for web vs mobile, the cleanest split is a separate `mobile_refresh_token_expire_seconds` setting + `MobileAuthService.refresh_access_token()` using that one; today they share the value.
  - **Verification after bump**: log in on the iOS app, then check the JWT's `exp` claim on the refresh token returned by `/mobile/auth/login` — should equal `now + 2592000` (30d). Optionally, verify a 25-hour-old refresh token still works by manually setting the `RefreshToken.CreatedAt` back in the DB and hitting `/mobile/auth/refresh`.

- [ ] **Mobile refresh-token rotation race on app SIGKILL (lower priority).** `POST /mobile/auth/refresh` rotates: server creates new refresh token `R2`, invalidates `R1`, returns `R2`. iOS persists `R2` to Keychain after the response lands ([AuthenticatedSession.swift:89-91](https://github.com/chriszubrod/build.one.ios/blob/master/BuildOne/Services/BuildOneAPI/AuthenticatedSession.swift#L89-L91)). If the app is killed between (a) the server's rotation and (b) the Keychain write committing, the user is locked out — next launch sends `R1`, server says "rotated out", iOS surfaces `tokenRefreshFailed` → critical alert → login screen. **In production** this requires a crash / OOM kill / user force-quit during the ~10-50ms refresh window — rare but real. **In dev** it's far more common: Xcode's Stop button is SIGKILL, and we hit this during the iOS work pass (session was observed routed to login on 2026-06-02 after heavy rebuild iteration). Two possible fixes, both server-side:
  1. **Grace window (preferred)**: keep `R1` valid for ~5 minutes after `R2` is issued so a client that didn't persist `R2` can still authenticate with `R1` once. Requires `RefreshToken.RotatedFromId` + `RotatedAt` columns + a `RevokeOldRotation` background sweep that purges grace-window tokens after the window. Standard OAuth pattern.
  2. **No-op no-rotate response**: detect "already rotated" specifically (vs "expired") and return the previously-issued `R2` instead of 401, with idempotency keyed off the request body. Simpler but breaks the rotation invariant (caller could replay forever within the grace).
  - Option 1 is the correct shape long-term; defer until refresh failures appear in prod logs OR the iOS team builds a dev-loop tool that needs it.

- [ ] **Device-token rebind cleanup on `/mobile/device-token/register` (push leak prevention).** Today the iOS app calls `POST /mobile/device-token/deactivate` during `userInitiatedLogout` to clean up the `(deviceToken → userId)` mapping server-side. That call requires the just-logged-out user's auth token to authenticate, and silently fails if the network is down at the moment Sign Out is tapped (logged via `os.Logger` subsystem `one.build.app` category `push` from 2026-06-02). The premise that this is "non-fatal because the token expires naturally" is **wrong** — APNs device tokens don't expire on a schedule (they're stable per device + app), and the server-side mapping persists indefinitely. Without cleanup, when User B logs in on the same device, the dormant `(token, UserA)` mapping can leak pushes addressed to UserA into UserB's notification center once the push sender ships. **Fix shape**: `/mobile/device-token/register` should atomically clear any prior `(token, *)` mappings before/while inserting the new `(token, newUser)` row. One device-token has at most one user at a time; the server is the only place that has authoritative knowledge to enforce that invariant (the client can't retry post-logout because its auth context is already cleared). Touches `dbo.DeviceToken` (likely a new `UpsertDeviceTokenAndClearPriorMappings` sproc or an `INSTEAD OF INSERT` trigger) and possibly the `entities/device_token/` service to thread the cleanup through. Higher priority **before** the push sender ships — once notifications start going out, this becomes a privacy bug. Today (no sender) the failure surface is purely latent.

## Expense coding cockpit (U-005) — Pass-2 deferrals (logged 2026-07-14, Phase A)

- [ ] **[U-005][efficiency] Set-based seed for `get_expense_coding_queue`.** The first-ever queue load lazily seeds a `dbo.ExpenseCodingItem` per unseeded 58999 line via a per-row `upsert_from_queue` (each ~3 sproc round-trips incl. `_resolve_vendor_id`), then re-reads the whole queue → up to ~1,300 round-trips on the cold ~440-row first load. **Cold path / one-time**: steady-state inflow is ~1–2/day and `needs_reseed` short-circuits once every line has a row.
