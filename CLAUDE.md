## Project Conventions

- **Entity pattern**: `entities/{name}/` with `api/`, `business/`, `persistence/`, `sql/` sub-packages. `web/` subpackages are gone as of Wave E5 — React owns the UI.
- **SQL**: All DB access via stored procedures (pyodbc). Migrations run with `python scripts/run_sql.py path/to/file.sql`
- **Stored procedure single-source-of-truth (standing rule; conversion in progress).** The entity's base SQL file (`entities/{name}/sql/dbo.{name}.sql`) is the **target** canonical home for a sproc — a migration must not carry a duplicate body of a sproc its entity's base file defines; change the base file and apply it. Base files use `CREATE OR ALTER` and get re-run routinely; an unapplied migration leaves prod sprocs behind the Python repo layer (pyodbc parameter errors). Incident: `CreateInvoiceLineItem` lacked `@EmployeeLaborLineItemId` in prod through WVA-17/WVA-18 (2026-07) because migration `001_2026_05_27_employee_labor_source.sql` was neither applied nor ported; base file made canonical 2026-07-06. New sproc params take `= NULL` defaults so older callers stay safe. **Converted:** `time_entry` (U-045, 2026-07-16) and `role_module` (U-048, 2026-07-16) — each entity's base file is provably the sole copy of its sprocs, enforced by `tests/test_sproc_single_source.py` (parametrized per entity; add a row to its table when you convert the next one); build order in each entity's `sql/README.md`. **EXCEPTION — the `dbo.UserCanAccess*` access-UDF family is NOT entity-homed (U-051, 2026-07-16).** All five (`TimeEntry`, `Project`, `Bill`, `BillCredit`, `Expense`) live in **`shared/sql/dbo.access_udfs.sql`**, build order in `shared/sql/README.md`. Do **not** fold `UserCanAccessBill` into `dbo.bill.sql` when you convert `bill` (nor Project/BillCredit/Expense into theirs) — those four are `WITH SCHEMABINDING` against another entity package's table whose file FKs back, so an entity-local home is a from-scratch build **cycle** (`dbo.userproject.sql` → `FK_UserProject_Project`; `dbo.bill_line_item.sql` → `FK_BillLineItem_Bill`), not a preference. `UserCanAccessTimeEntry` is the exception: it has **no** cycle (a plain ordering dep on UserProject), which is why U-045 could home it in the time_entry base file — U-051 relocated it anyway so the family sits in one place under one guard, superseding that placement. The guard here is default-ON: any `UserCanAccess*` UDF defined outside the shared home fails the suite, so a 6th cannot silently escape. Note `SCHEMABINDING` binds **columns** — those four UDFs bind `{Project,Bill,BillCredit,Expense}.CreatedByUserId`, which is added by `scripts/migrations/gap2_created_by_user_id.sql`, not by the base `CREATE TABLE` blocks. **Watch the T-SQL batch trap when converting** (found in `dbo.rolemodule.sql`, U-048): a `CREATE PROCEDURE` body runs to the end of the **batch**, not to its matching `END` — so a trailing `IF NOT EXISTS … ALTER TABLE` constraint block with no `GO` before it gets silently swallowed **into the last sproc's body**, which both mutates that sproc on re-apply and means the constraint never executes. Verify every sproc batch is `GO`-terminated. **Not yet converted:** 35 other entities still have base sprocs duplicated in migrations (bill=10, contract_labor=10, user_role=9, …) — for those, the base file may be stale; **before redefining any sproc in a migration, diff the LIVE sproc's params (`sys.parameters`) against the base file** — if they differ, the base is stale and copying from it drops prior work. Incident 2026-07-15 (U-037): migration `015_status_read_id_tiebreak.sql` redefined 4 time-entry sprocs from a stale UNSCOPED base, dropping RBAC actor params → `/time-entries` 500 (SQL 8144), cross-user payroll-exposure risk; U-037/U-039 reconciled the base file before U-045 collapsed duplicate bodies into pointer stubs.
- **Concurrency**: SQL Server ROWVERSION columns with base64 encoding for transport
- **UI**: all UI is React (`build.one.web`). No server-rendered templates, no `templates/` directory. API routes return JSON only.
- **Workflow engine**: ProcessEngine for main entity CRUD; lightweight child entities use direct CRUD
- **Lazy imports**: Some services (e.g., BillService) use lazy imports in `__init__` to avoid circular deps — always use `self.*` instance attributes, not bare class names
- **Stored procedure NULL handling**: UPDATE sprocs unconditionally SET all columns. Use CASE WHEN guards for fields that should preserve existing values when NULL is passed
- **Stored procedure result-set discipline (pyodbc, learned 2026-06-11)**: mutation sprocs MUST `SET NOCOUNT ON` — bare DML before the final SELECT emits row-count tokens as the first "result" and `cursor.fetchone()` raises "No results. Previous SQL was not a query" (single-statement `INSERT/UPDATE ... OUTPUT` sprocs survive by accident; anything multi-statement does not). And NEVER `ROLLBACK TRANSACTION` inside a sproc — pyodbc runs autocommit-off, so the in-proc rollback zeroes the implicit outer transaction and SQL Server raises error 266 ("Transaction count after EXECUTE…"). Use the validate-first / always-COMMIT shape: take UPDLOCK locks, validate, mutate only when valid, COMMIT unconditionally, and signal conflicts with an empty final result set (or `COMMIT` the untouched txn then `RAISERROR`). The Budget sprocs (entities/budget*/sql/) are the canonical examples.

- **QBO pull must not clobber a human-edited document number ("rule of three", U-027)**: the pull connectors derive the document number as `qbo.doc_number or f"QBO-{qbo_id}"`. On the **UPDATE path** never pass that derived value straight through — a scheduler re-pull would revert a local human correction within one tick (KI-42). Applied to **Bill.bill_number**, **BillCredit.credit_number**, **Expense.reference_number** (U-024), and **Invoice.invoice_number** (U-034 — with the fingerprint + provenance adopt guard described below). **CREATE paths stay unchanged** (always the QBO-derived value). Any new QBO pull connector with a human-editable number must reuse this helper. **Caveat — do NOT blindly apply it to a connector whose lost-mapping recovery keys on the QBO-derived number.** `Invoice.invoice_number` was the tricky case (its gap-detect/adopt path `read_by_invoice_number_and_project_id` keyed on the QBO-derived number, so a naive preserve reintroduced the "46 phantom -N invoices" bug — U-027 deferred it). **Resolved (U-034):** the adopt path now falls back to a header **fingerprint** (total within `Decimal("0.01")` + txn_date + project) when the number lookup misses, and requires **QBO-line-provenance** (`_has_qbo_line_provenance` — the invoice carries ≥1 `InvoiceLineItemInvoiceLine` line-mapping) on BOTH adopt routes before binding — so a human-renamed, mapping-lost invoice is RE-ADOPTED (not phantom-duplicated) while a distinct **manual** invoice (which stays unmapped because Invoice→QBO push is disabled) is never wrongly adopted. Preserve is applied on both the plain UPDATE and the adopt UPDATE. Connector-only, no schema change. Bill/VendorCredit carry a narrower documented residual (their `(vendor, number)` create-dedup is only defeated if a mapping is abnormally lost while the entity persists). See TODO.md.
- **QBO pull "heal-don't-delete" on a mapping-exists-but-entity-read-empty ("rule of three" COMPLETE — U-022 customer→Project, U-029 purchase→Expense, U-031 Bill + VendorCredit)**: a pull connector must NEVER `mapping_repo.delete_by_id(...)` then fall through to CREATE when the mapping's bound entity reads empty — a *transient* empty-read then mints a DUPLICATE. (VendorCredit's flavor was subtler: it didn't delete the mapping, it silently fell through to its Step-3 CREATE on the empty read — same duplicate outcome, fixed the same way.) Re-resolve BEFORE mutating: **repoint the mapping in place** when a replacement binds (Project — it has an `update_by_id` sproc), or **when there is no repoint sproc** (Expense/Bill/BillCredit — their mapping repos have none, and no SQL migration was in scope) heal in place **only if a natural fingerprint re-binds the SAME mapped id** — `(reference_number, vendor)` for Expense (`ExpenseService.read_by_reference_number_and_vendor_public_id`), `(bill_number, vendor)` for Bill (`BillService.read_by_bill_number_and_vendor_public_id`), `(credit_number, vendor)` for BillCredit (`BillCreditService.read_by_credit_number_and_vendor_public_id`) — routing the number through `preserve_human_edited_ref` exactly like the happy path; on any other outcome record a critical `[qbo].[ReconciliationIssue]` (`orphaned_{purchase_expense,bill_bill,vendorcredit_billcredit}_mapping`, failure-isolated) and **RAISE, mutating nothing** — the caller's per-item handler skips it and the issue is the durable follow-up record. Never auto-recreate on an empty read. The same-id gate makes a wrong/duplicate row safe even when the fingerprint proc is not `TOP 1` (`id != mapping` → record+raise, never a wrong bind). NB the pull treats a raised `ValueError` as a *permanent skip* (watermark advances) — the in-tick fingerprint heal is the transient-recovery path, not "retry next tick". Applies to any new `sync_from_qbo_*`. Still OPEN (own units, need SQL / span connectors): a mapping-repoint sproc so a *different-id* fingerprint match can heal instead of raise, and lifting the heal + reconciliation-recorder shape into one shared QBO helper (three hand-copied recorders now — see TODO.md).

## External Integrations (QBO + MS) — Hardening Patterns (April 2026)

Both `integrations/intuit/qbo/` and `integrations/ms/` follow the same hardened shape after the April 2026 refactors; `integrations/box/` (live 2026-06-16) follows it too. Future external integrations must follow this pattern.

- **`base/` package per integration**: `client.py` (shared HTTP wrapper), `errors.py` (typed exception hierarchy with `is_retryable`), `locking.py` (`sp_getapplock` context manager), `logger.py` (`LoggerAdapter` that auto-injects `correlation_id` from a ContextVar), `retry.py` (RetryPolicy + `execute_with_retry` with backoff + jitter + Retry-After), `correlation.py` (ContextVar-backed correlation + idempotency key), `idempotency.py` (UUID generation / validation).
- **Durable outbox pattern**: every external write enqueues a row in `[qbo].[Outbox]` / `[ms].[Outbox]` / `[box].[Outbox]` instead of calling the external API inline. Drain is triggered by the `build.one.scheduler` Azure Function App (see "Scheduler Architecture" below). **As of 2026-06-15 each integration drains on its own 30s timer + endpoint** — `/api/v1/admin/outbox/drain/qbo`, `/api/v1/admin/outbox/drain/ms`, `/api/v1/admin/box/drain` — so a slow drain in one can't head-of-line-block another (the old combined `/api/v1/admin/outbox/drain` remains as a deprecated manual-fallback alias). The handler claims rows via `ClaimNextPending*Outbox` (UPDLOCK+READPAST) and dispatches through per-Kind handlers. 5 failed attempts → dead-letter. Policy C debounce coalesces rapid edits via `ReadyAfter` column. (In-process APScheduler fallback exists in `shared/scheduler.py` gated on `ENABLE_SCHEDULER=true`; currently disabled in prod.)
- **Outbox workers self-declare system intent (2026-05-14).** `QboOutboxWorker._process` and `MsOutboxWorker._process` wrap their body with `set_authz_context(is_system_admin=True)` (save → set → restore) so guarded service reads (`BillService.read_*`, `ExpenseService.read_*`, etc.) bypass the per-row access UDFs. Required because the 2026-05-12 access-guard tightening removed the legacy `current_user_id is None → bypass` branch — without explicit system intent, every drain dead-letters as `EntityNotAccessibleError` on attempt 1. The HTTP drain endpoint also sets the context via `_require_drain_secret`; the worker self-defense is a belt-and-suspenders against future entry points. CLI sync scripts (`scripts/sync_qbo_*.py`) carry the same belt: they call `assert_cli_system_admin()` from `scripts/sync_helper.py` under `__main__`. New sync entry points must follow this pattern. See `feedback_outbox_authz_boundary.md`.
- **Write gates**: `ALLOW_QBO_WRITES=true` and `ALLOW_MS_WRITES=true` must be explicitly set (case-insensitive `"true"`) in prod App Service Application Settings. Default-deny — a fresh checkout cannot accidentally push to prod. The gate applies at both the shared HTTP client level AND at `*OutboxService.enqueue()`.
- **Scheduler jobs must be async-wrapped.** Only relevant if `ENABLE_SCHEDULER=true` (fallback mode). `shared/scheduler.py` registers every job as `async def wrapper()` that awaits `asyncio.to_thread(sync_fn)`. Blocking I/O (httpx, pyodbc) runs on the default thread pool; the FastAPI event loop stays free. Synchronous jobs registered on `AsyncIOScheduler` would block ALL user requests during the tick — do not regress this.
- **Token storage encrypted at rest**: `QboAuthRepository` and `MsAuthRepository` both wrap access/refresh tokens with `shared.encryption.encrypt_sensitive_data()` on write and `decrypt_if_encrypted()` on read. Requires `ENCRYPTION_KEY` env var in prod; self-heals plaintext rows on next write (no migration required).
- **Dead-letter recovery**: `scripts/retry_ms_outbox_dead_letters.py` (and future QBO equivalent) resets rows back to `pending`. Dry-run by default; `--apply` to mutate; `--kind` to filter.
- **Reconciliation**: `[qbo].[ReconciliationIssue]` / `[ms].[ReconciliationIssue]` capture drift + dead-letter escalations. Excel-kind MS dead-letters escalate to **critical** severity so they aren't silent. Daily reconciliation jobs detect "locally complete but not in external system" drift.
- **Completion pipelines enqueue, never call external APIs inline.** `BillService.complete_bill` / `ExpenseService.complete_expense` finalize the entity locally, then enqueue outbox rows for SharePoint upload, Excel sync, QBO push. The API returns 202 immediately; side effects drain asynchronously within ~5-30s.
- **Runbook discipline**: every external failure mode documented in `docs/runbooks/*.md` with Symptom / Severity / Diagnosis / Recovery / Verification / Prevention sections. Follow the existing runbook format when adding new ones.
- **Box integration (`integrations/box/`, LIVE 2026-06-16).** Third integration on this pattern: CCG **as-user** auth (mints AS Chris's managed user `BOX_AS_USER_ID=31760447449` — no refresh tokens, no per-folder service-account collaboration), `[box].[Outbox]` + own `drain_box_outbox` timer, `ALLOW_BOX_WRITES` gate. Two write pipelines, both additive to the MS/SharePoint equivalents (both fire on completion):
  - **Excel-in-Box**: `[box].[ProjectWorkbook]` maps Project → Box `.xlsx`; completion enqueues `update_box_excel`; drain downloads → openpyxl-edits the DETAILS tab (col-Z line-item public_id idempotency) → uploads a new version (Box has no cell API). Worksheet tab name is per-project (`DETAILS` / `Tracking Budget DETAILS` / `DETAIL`). WOPI-lock → defer-while-open. `_sanitize_workbook_bytes` strips `#N/A` print-title defined names openpyxl rejects.
  - **Document push**: `[box].[ProjectFolder]` maps **(Project, DocClass)** → Box folder. **DocClass routing**: bill / expense / bill_credit attachments → `'invoices'` (the project's `14 - Invoices` folder); invoice packets → `'draw_requests'` (`15 - Draw Requests`) — mirrors the SharePoint `[ms].[DriveItemProjectModule]` per-module split. A Box folder may be **SHARED** across sub-unit projects (sub-units file under the parent property) → `BoxFolderId` is NOT unique on `[box].[ProjectFolder]`; only `(ProjectId, DocClass)` is. `push_blob_to_box` uses deterministic identity-embedded filenames + 409→re-version + sha1 verify.
  - Map/unmap via `POST /box/{map-project,unmap-project,map-workbook}` (QBO_SYNC-gated) or scripts/SQL — **no web UI** (Chris's call; operate like QBO/MS). **Forward-only**: only NEW completions push; do NOT back-fill old entities (no dedup vs hand-filed docs/rows). Per-project mappings gate the blast radius — unmapped projects skip cleanly.

## Scheduler Architecture (April 2026)

Recurring jobs (outbox drain, QBO pulls, daily reconciles) live in the sibling `build.one.scheduler` repo — an Azure Function App (**Flex Consumption**, Python 3.11) that POSTs to admin endpoints on the API.

- **Why separate**: in-process APScheduler on `-w 1/2` stole threadpool slots from user requests. Moving it out eliminates contention and lets the API run at `-w 2` comfortably.
- **Flex `alwaysReady` is required for timer reliability (2026-05-06).** Without it the host scales to zero and timer triggers silently miss ticks (App Insights also goes silent because nothing's running to emit telemetry — same root cause for both symptoms). One always-ready instance keyed off `function:drain_outbox=1` keeps the Python worker continuously running; all 18 timers in the app share it. Set via `az functionapp scale config always-ready set --name build-one-scheduler --resource-group buildone_group --settings function:drain_outbox=1`. Costs ~$5–15/mo. **Do NOT use `function_app=1`** — Flex API rejects it; valid groups are only `$http`, `blob`, `durable`, or per-function `function:<name>`.
- **Flex Consumption gotcha (2026-05-06).** Setting `AzureWebJobs.<funcName>.Disabled=true` on a Flex Consumption Function App de-registers that function from host discovery, and setting it back to `false` (or deleting the app-setting) does NOT cause re-registration even after stop+start. Other timers keep firing; the disabled-then-re-enabled one stays missing from `az functionapp function list`. **Don't use that flag on Flex.** Recovery is `func azure functionapp publish build-one-scheduler --python` to force re-discovery. See `docs/runbooks/deploy-restart-timing.md` for the full recovery procedure. (Symptom: `process_email_inbox` was paused mid-session via the Disabled flag; un-pausing required a redeploy.)
- **Admin endpoints** (see `shared/api/admin.py`): all require `X-Drain-Secret` header.
  - `POST /api/v1/admin/outbox/drain/qbo` — drains QBO outbox (30s); `POST /api/v1/admin/outbox/drain/ms` — drains MS outbox (30s); `POST /api/v1/admin/box/drain` — drains Box outbox (30s). Per-integration since 2026-06-15. `POST /api/v1/admin/outbox/drain` is the DEPRECATED combined alias (manual fallback, no timer).
  - `POST /api/v1/admin/sync/qbo/{entity}` — one of bill/invoice/purchase/vendorcredit/vendor/customer/item/account/term/company_info
  - `POST /api/v1/admin/reconcile/qbo` — daily
  - `POST /api/v1/admin/reconcile/ms` — daily
  - `POST /api/v1/admin/bill-folder/tick` — claim-and-process one queued PDF (30s cadence; scheduler inner-loops up to 4 min)
  - `POST /api/v1/admin/bill-folder/enumerate` — scan `/source` and enqueue new PDFs with hour-window dedup (5m cadence)
  - `POST /api/v1/admin/email/poll` — poll the shared invoice mailbox for tagged messages (5m cadence)
  - `POST /api/v1/admin/email/process_one` — claim oldest pending EmailMessage and kick off email_specialist agent run (1m cadence, inner-drain up to 4 min)
  - `POST /api/v1/admin/email/recover_stuck` — sweep orphaned EmailMessage rows (`processing` + NULL AgentSessionId) back to `pending` + time out long-running AgentSessions (5m cadence at :01,:06,:11,…). Each EmailMessage carries a `ProcessingResetCount`; over-budget rows dead-letter to `failed`. Closes the gap that `ClaimNextPendingEmailMessage` commits before `start_run` inserts the AgentSession (a crash between those two steps orphans the row).
  - `POST /api/v1/admin/email/extract/{public_id}` — DRAIN_SECRET-gated DI extraction trigger for verification (the email agent uses the RBAC-gated `/email-attachments/{public_id}/extract` instead)
  - `POST /api/v1/admin/time-entry/daily-digest` — emails each worker a summary of a day's time entries + logs for correctness confirmation (default: yesterday in `BUSINESS_TIMEZONE`; `?work_date=YYYY-MM-DD` to re-run a day). Once-daily (`time_entry_daily_digest` timer, 13:00 UTC ≈ 8am Central). Enqueues one MS-outbox `send_mail` per worker; idempotent per (worker, day) via a deterministic uuid5 `EntityPublicId`. Honors `TIME_ENTRY_DIGEST_MODE` (off|draft|send — ships off) + `ALLOW_MS_WRITES`.
- **Secret**: `DRAIN_SECRET` env var on both App Service + Function App. Never commit. Rotate if leaked.
- **Rollback to in-process scheduler**: set `ENABLE_SCHEDULER=true` on App Service + restart. Safe — `sp_getapplock` prevents Function + in-process from double-draining.
- **Fail-closed auth**: admin endpoints return 503 when `DRAIN_SECRET` is unset on the server, 401 on mismatch.

## App Service Config — Required Production Settings

- **Always On = true**: critical. Without it, the container gets unloaded after ~20 min of inactivity and cold starts take 10–15 seconds. Enable via `az webapp config set --name <app> --resource-group <rg> --always-on true`.
- **`-w 2`**: configured in `startup.sh`. Two gunicorn/uvicorn workers on B2. Fits in memory now that APScheduler is out of the web process. `-w 4` is feasible but unnecessary at current load.
- **Env vars** that must exist on App Service: `ENABLE_SCHEDULER=false` (in-process scheduler disabled), `DRAIN_SECRET` (matches Function App), `ALLOW_QBO_WRITES=true`, `ALLOW_MS_WRITES=true`, `ALLOW_EXPENSE_RECODE_WRITES=false` (U-005 expense-coding recode gate; default off — flip to true only at Gate-P2 go-live), `ENCRYPTION_KEY`, plus DB creds. **Email-agent additions:** `email_agent_username` + `email_agent_password` for the new agent's login; `azure_document_intelligence_endpoint` + `azure_document_intelligence_key` (+ optional `azure_document_intelligence_api_version`, defaults to `2024-11-30`); `invoice_inbox_email` for the shared mailbox the poll service targets. `azure_encryption_key` is an optional dev-only override that takes precedence over `encryption_key` so a developer can decrypt prod-encrypted DB rows locally without touching the dev-only key (see `shared/encryption.py::_get_encryption_key`). **Time-entry digest additions:** `TIME_ENTRY_DIGEST_MODE` (off|draft|send; default `off`) is the kill switch + mode for the daily per-worker time-entry digest; optional `BUSINESS_TIMEZONE` (default `America/Chicago`, used to compute "yesterday"; clock times are rendered verbatim as stored local wall-clock — no conversion) and `TIME_ENTRY_DIGEST_BCC` (archive BCC; falls back to `invoice_inbox_email`).
