## Project Conventions

- **Entity pattern**: `entities/{name}/` with `api/`, `business/`, `persistence/`, `sql/` sub-packages. `web/` subpackages are gone as of Wave E5 ŌĆö React owns the UI.
- **SQL**: All DB access via stored procedures (pyodbc). Migrations run with `python scripts/run_sql.py path/to/file.sql`
- **Migrations that redefine sprocs MUST port the change into the entity's base SQL file in the same commit.** Base files use `CREATE OR ALTER` and get re-run routinely ŌĆö an unported migration is silently reverted by the next base re-run, and an unapplied migration leaves prod sprocs behind the Python repo layer (pyodbc parameter errors). Incident: `CreateInvoiceLineItem` lacked `@EmployeeLaborLineItemId` in prod through WVA-17/WVA-18 (2026-07) because migration `001_2026_05_27_employee_labor_source.sql` was neither applied nor ported; base file made canonical 2026-07-06. New sproc params take `= NULL` defaults so older callers stay safe.
- **Concurrency**: SQL Server ROWVERSION columns with base64 encoding for transport
- **UI**: all UI is React (`build.one.web`). No server-rendered templates, no `templates/` directory. API routes return JSON only.
- **Workflow engine**: ProcessEngine for main entity CRUD; lightweight child entities use direct CRUD
- **Lazy imports**: Some services (e.g., BillService) use lazy imports in `__init__` to avoid circular deps ŌĆö always use `self.*` instance attributes, not bare class names
- **Stored procedure NULL handling**: UPDATE sprocs unconditionally SET all columns. Use CASE WHEN guards for fields that should preserve existing values when NULL is passed
- **Stored procedure result-set discipline (pyodbc, learned 2026-06-11)**: mutation sprocs MUST `SET NOCOUNT ON` ŌĆö bare DML before the final SELECT emits row-count tokens as the first "result" and `cursor.fetchone()` raises "No results. Previous SQL was not a query" (single-statement `INSERT/UPDATE ... OUTPUT` sprocs survive by accident; anything multi-statement does not). And NEVER `ROLLBACK TRANSACTION` inside a sproc ŌĆö pyodbc runs autocommit-off, so the in-proc rollback zeroes the implicit outer transaction and SQL Server raises error 266 ("Transaction count after EXECUTEŌĆ”"). Use the validate-first / always-COMMIT shape: take UPDLOCK locks, validate, mutate only when valid, COMMIT unconditionally, and signal conflicts with an empty final result set (or `COMMIT` the untouched txn then `RAISERROR`). The Budget sprocs (entities/budget*/sql/) are the canonical examples.
- **Auto-save**: Bill and Expense edit pages debounce saves at 300ms. Any action that depends on persisted state (Complete, Delete) must flush (`await autoSave()`) or guard (`isSaving = true`) against pending auto-saves before sending the request
- **RBAC chain**: User ŌåÆ UserRole ŌåÆ Role ŌåÆ RoleModule ŌåÆ Module. Role entity is the core ŌĆö UserRole and RoleModule are join tables. Role assignment is managed inline on the User Profile page. ## External Integrations (QBO + MS) ŌĆö Hardening Patterns (April 2026)

Both `integrations/intuit/qbo/` and `integrations/ms/` follow the same hardened shape after the April 2026 refactors; `integrations/box/` (live 2026-06-16) follows it too. Future external integrations must follow this pattern.

- **`base/` package per integration**: `client.py` (shared HTTP wrapper), `errors.py` (typed exception hierarchy with `is_retryable`), `locking.py` (`sp_getapplock` context manager), `logger.py` (`LoggerAdapter` that auto-injects `correlation_id` from a ContextVar), `retry.py` (RetryPolicy + `execute_with_retry` with backoff + jitter + Retry-After), `correlation.py` (ContextVar-backed correlation + idempotency key), `idempotency.py` (UUID generation / validation).
- **Durable outbox pattern**: every external write enqueues a row in `[qbo].[Outbox]` / `[ms].[Outbox]` / `[box].[Outbox]` instead of calling the external API inline. Drain is triggered by the `build.one.scheduler` Azure Function App (see "Scheduler Architecture" below). **As of 2026-06-15 each integration drains on its own 30s timer + endpoint** ŌĆö `/api/v1/admin/outbox/drain/qbo`, `/api/v1/admin/outbox/drain/ms`, `/api/v1/admin/box/drain` ŌĆö so a slow drain in one can't head-of-line-block another (the old combined `/api/v1/admin/outbox/drain` remains as a deprecated manual-fallback alias). The handler claims rows via `ClaimNextPending*Outbox` (UPDLOCK+READPAST) and dispatches through per-Kind handlers. 5 failed attempts ŌåÆ dead-letter. Policy C debounce coalesces rapid edits via `ReadyAfter` column. (In-process APScheduler fallback exists in `shared/scheduler.py` gated on `ENABLE_SCHEDULER=true`; currently disabled in prod.)
- **Outbox workers self-declare system intent (2026-05-14).** `QboOutboxWorker._process` and `MsOutboxWorker._process` wrap their body with `set_authz_context(is_system_admin=True)` (save ŌåÆ set ŌåÆ restore) so guarded service reads (`BillService.read_*`, `ExpenseService.read_*`, etc.) bypass the per-row access UDFs. Required because the 2026-05-12 access-guard tightening removed the legacy `current_user_id is None ŌåÆ bypass` branch ŌĆö without explicit system intent, every drain dead-letters as `EntityNotAccessibleError` on attempt 1. The HTTP drain endpoint also sets the context via `_require_drain_secret`; the worker self-defense is a belt-and-suspenders against future entry points. CLI sync scripts (`scripts/sync_qbo_*.py`) carry the same belt: they call `assert_cli_system_admin()` from `scripts/sync_helper.py` under `__main__`. New sync entry points must follow this pattern. See `feedback_outbox_authz_boundary.md`.
- **Write gates**: `ALLOW_QBO_WRITES=true` and `ALLOW_MS_WRITES=true` must be explicitly set (case-insensitive `"true"`) in prod App Service Application Settings. Default-deny ŌĆö a fresh checkout cannot accidentally push to prod. The gate applies at both the shared HTTP client level AND at `*OutboxService.enqueue()`.
- **Scheduler jobs must be async-wrapped.** Only relevant if `ENABLE_SCHEDULER=true` (fallback mode). `shared/scheduler.py` registers every job as `async def wrapper()` that awaits `asyncio.to_thread(sync_fn)`. Blocking I/O (httpx, pyodbc) runs on the default thread pool; the FastAPI event loop stays free. Synchronous jobs registered on `AsyncIOScheduler` would block ALL user requests during the tick ŌĆö do not regress this.
- **Token storage encrypted at rest**: `QboAuthRepository` and `MsAuthRepository` both wrap access/refresh tokens with `shared.encryption.encrypt_sensitive_data()` on write and `decrypt_if_encrypted()` on read. Requires `ENCRYPTION_KEY` env var in prod; self-heals plaintext rows on next write (no migration required).
- **Dead-letter recovery**: `scripts/retry_ms_outbox_dead_letters.py` (and future QBO equivalent) resets rows back to `pending`. Dry-run by default; `--apply` to mutate; `--kind` to filter.
- **Reconciliation**: `[qbo].[ReconciliationIssue]` / `[ms].[ReconciliationIssue]` capture drift + dead-letter escalations. Excel-kind MS dead-letters escalate to **critical** severity so they aren't silent. Daily reconciliation jobs detect "locally complete but not in external system" drift.
- **Completion pipelines enqueue, never call external APIs inline.** `BillService.complete_bill` / `ExpenseService.complete_expense` finalize the entity locally, then enqueue outbox rows for SharePoint upload, Excel sync, QBO push. The API returns 202 immediately; side effects drain asynchronously within ~5-30s.

Two write pipelines, both additive to the MS/SharePoint equivalents (both fire on completion):
  - **Excel-in-Box**: `[box].[ProjectWorkbook]` maps Project ŌåÆ Box `.xlsx`; completion enqueues `update_box_excel`; drain downloads ŌåÆ openpyxl-edits the DETAILS tab (col-Z line-item public_id idempotency) ŌåÆ uploads a new version (Box has no cell API). Worksheet tab name is per-project (`DETAILS` / `Tracking Budget DETAILS` / `DETAIL`). WOPI-lock ŌåÆ defer-while-open. `_sanitize_workbook_bytes` strips `#N/A` print-title defined names openpyxl rejects.
  - **Document push**: `[box].[ProjectFolder]` maps **(Project, DocClass)** ŌåÆ Box folder. **DocClass routing**: bill / expense / bill_credit attachments ŌåÆ `'invoices'` (the project's `14 - Invoices` folder); invoice packets ŌåÆ `'draw_requests'` (`15 - Draw Requests`) ŌĆö mirrors the SharePoint `[ms].[DriveItemProjectModule]` per-module split. A Box folder may be **SHARED** across sub-unit projects (sub-units file under the parent property) ŌåÆ `BoxFolderId` is NOT unique on `[box].[ProjectFolder]`; only `(ProjectId, DocClass)` is. `push_blob_to_box` uses deterministic identity-embedded filenames + 409ŌåÆre-version + sha1 verify.
  - Map/unmap via `POST /box/{map-project,unmap-project,map-workbook}` (QBO_SYNC-gated) or scripts/SQL ŌĆö **no web UI** (Chris's call; operate like QBO/MS). **Forward-only**: only NEW completions push; do NOT back-fill old entities (no dedup vs hand-filed docs/rows). Per-project mappings gate the blast radius ŌĆö unmapped projects skip cleanly.

## Scheduler Architecture (April 2026)

Recurring jobs (outbox drain, QBO pulls, daily reconciles) live in the sibling `build.one.scheduler` repo ŌĆö an Azure Function App (**Flex Consumption**, Python 3.11) that POSTs to admin endpoints on the API.

- **Why separate**: in-process APScheduler on `-w 1/2` stole threadpool slots from user requests. Moving it out eliminates contention and lets the API run at `-w 2` comfortably.
- **Flex `alwaysReady` is required for timer reliability (2026-05-06).** Without it the host scales to zero and timer triggers silently miss ticks (App Insights also goes silent because nothing's running to emit telemetry ŌĆö same root cause for both symptoms). One always-ready instance keyed off `function:drain_outbox=1` keeps the Python worker continuously running; all 18 timers in the app share it. Set via `az functionapp scale config always-ready set --name build-one-scheduler --resource-group buildone_group --settings function:drain_outbox=1`. Costs ~$5ŌĆō15/mo. **Do NOT use `function_app=1`** ŌĆö Flex API rejects it; valid groups are only `$http`, `blob`, `durable`, or per-function `function:<name>`.
- **Flex Consumption gotcha (2026-05-06).** Setting `AzureWebJobs.<funcName>.Disabled=true` on a Flex Consumption Function App de-registers that function from host discovery, and setting it back to `false` (or deleting the app-setting) does NOT cause re-registration even after stop+start. Other timers keep firing; the disabled-then-re-enabled one stays missing from `az functionapp function list`. **Don't use that flag on Flex.** Recovery is `func azure functionapp publish build-one-scheduler --python` to force re-discovery. See `docs/runbooks/deploy-restart-timing.md` for the full recovery procedure. (Symptom: `process_email_inbox` was paused mid-session via the Disabled flag; un-pausing required a redeploy.)
- **Admin endpoints** (see `shared/api/admin.py`): all require `X-Drain-Secret` header.
  - `POST /api/v1/admin/outbox/drain/qbo` ŌĆö drains QBO outbox (30s); `POST /api/v1/admin/outbox/drain/ms` ŌĆö drains MS outbox (30s); `POST /api/v1/admin/box/drain` ŌĆö drains Box outbox (30s). Per-integration since 2026-06-15. `POST /api/v1/admin/outbox/drain` is the DEPRECATED combined alias (manual fallback, no timer).
  - `POST /api/v1/admin/sync/qbo/{entity}` ŌĆö one of bill/invoice/purchase/vendorcredit/vendor/customer/item/account/term/company_info
  - `POST /api/v1/admin/reconcile/qbo` ŌĆö daily
  - `POST /api/v1/admin/reconcile/ms` ŌĆö daily
  - `POST /api/v1/admin/bill-folder/tick` ŌĆö claim-and-process one queued PDF (30s cadence; scheduler inner-loops up to 4 min)
  - `POST /api/v1/admin/bill-folder/enumerate` ŌĆö scan `/source` and enqueue new PDFs with hour-window dedup (5m cadence)
  - `POST /api/v1/admin/email/poll` ŌĆö poll the shared invoice mailbox for tagged messages (5m cadence)
  - `POST /api/v1/admin/email/process_one` ŌĆö claim oldest pending EmailMessage and kick off email_specialist agent run (1m cadence, inner-drain up to 4 min)
  - `POST /api/v1/admin/email/recover_stuck` ŌĆö sweep orphaned EmailMessage rows (`processing` + NULL AgentSessionId) back to `pending` + time out long-running AgentSessions (5m cadence at :01,:06,:11,ŌĆ”). Each EmailMessage carries a `ProcessingResetCount`; over-budget rows dead-letter to `failed`. Closes the gap that `ClaimNextPendingEmailMessage` commits before `start_run` inserts the AgentSession (a crash between those two steps orphans the row).
  - `POST /api/v1/admin/email/extract/{public_id}` ŌĆö DRAIN_SECRET-gated DI extraction trigger for verification (the email agent uses the RBAC-gated `/email-attachments/{public_id}/extract` instead)
  - `POST /api/v1/admin/time-entry/daily-digest` ŌĆö emails each worker a summary of a day's time entries + logs for correctness confirmation (default: yesterday in `BUSINESS_TIMEZONE`; `?work_date=YYYY-MM-DD` to re-run a day). Once-daily (`time_entry_daily_digest` timer, 13:00 UTC Ōēł 8am Central). Enqueues one MS-outbox `send_mail` per worker; idempotent per (worker, day) via a deterministic uuid5 `EntityPublicId`. Honors `TIME_ENTRY_DIGEST_MODE` (off|draft|send ŌĆö ships off) + `ALLOW_MS_WRITES`.
- **Secret**: `DRAIN_SECRET` env var on both App Service + Function App. Never commit. Rotate if leaked.
- **Rollback to in-process scheduler**: set `ENABLE_SCHEDULER=true` on App Service + restart. Safe ŌĆö `sp_getapplock` prevents Function + in-process from double-draining.
- **Fail-closed auth**: admin endpoints return 503 when `DRAIN_SECRET` is unset on the server, 401 on mismatch.

## App Service Config ŌĆö Required Production Settings

- **Always On = true**: critical. Without it, the container gets unloaded after ~20 min of inactivity and cold starts take 10ŌĆō15 seconds. Enable via `az webapp config set --name <app> --resource-group <rg> --always-on true`.
- **`-w 2`**: configured in `startup.sh`. Two gunicorn/uvicorn workers on B2. Fits in memory now that APScheduler is out of the web process. `-w 4` is feasible but unnecessary at current load.
- **Env vars** that must exist on App Service: `ENABLE_SCHEDULER=false` (in-process scheduler disabled), `DRAIN_SECRET` (matches Function App), `ALLOW_QBO_WRITES=true`, `ALLOW_MS_WRITES=true`, `ENCRYPTION_KEY`, plus DB creds. **Email-agent additions:** `email_agent_username` + `email_agent_password` for the new agent's login; `azure_document_intelligence_endpoint` + `azure_document_intelligence_key` (+ optional `azure_document_intelligence_api_version`, defaults to `2024-11-30`); `invoice_inbox_email` for the shared mailbox the poll service targets. `azure_encryption_key` is an optional dev-only override that takes precedence over `encryption_key` so a developer can decrypt prod-encrypted DB rows locally without touching the dev-only key (see `shared/encryption.py::_get_encryption_key`). **Time-entry digest additions:** `TIME_ENTRY_DIGEST_MODE` (off|draft|send; default `off`) is the kill switch + mode for the daily per-worker time-entry digest; optional `BUSINESS_TIMEZONE` (default `America/Chicago`, used to compute "yesterday"; clock times are rendered verbatim as stored local wall-clock ŌĆö no conversion) and `TIME_ENTRY_DIGEST_BCC` (archive BCC; falls back to `invoice_inbox_email`).
