## iOS service architecture review (v0.2.0+)

- [ ] **CoreData reads should happen on background contexts, not the main actor.** Paired with the lazy-cache refactor below. Today every service's cache-populate happens on `@MainActor` because `AppEnvironment` is `@MainActor @Observable` and every service init runs there. Cumulative 30 services × 30ms fetch = ~900ms of main-thread blocking before LoginView renders. Refactor: use `NSManagedObjectContext.perform { ... }` on a background context for the fetch, hop back to main actor with the results for UI binding. Do this at the same time as the lazy-cache refactor below — they touch the same code.
- [ ] **Move cache pre-populate out of service `init` to first-access.** Today every service in iOS `AppEnvironment.init` does a synchronous `CDX.fetchAll(in: coreData.viewContext)` to pre-populate its in-memory state. With 16 services on `@MainActor` this is fine (~100-400ms cold launch), but the cost is linear in service count + cache size. At v0.2 with 30+ services and bigger caches, this crosses the user-perceptible 1s threshold and starts blocking the LoginView from appearing. **Fix**: convert each service's init-time `fetchAll` into a `private func ensureCachePopulated()` called lazily at the top of any `loadX()` / computed-property read. Init becomes ~zero cost; cache pre-populate happens when the relevant view actually appears. Field workers who never touch Bills/Expenses/Invoices stop paying for those caches every cold launch. ~30 min per service when already touching the file. Zero behavior change for end users — they only pay for caches they navigate to.
- [ ] **Don't introduce a DI framework yet.** For 30 services, plain init injection in `AppEnvironment` is still cleaner than Swinject/Factory overhead. Revisit at ~100 services or when cold-launch construction exceeds 2s after the lazy-cache fix above.
- [ ] **Defer feature-container split until forced.** Splitting `AppEnvironment` into `CoreEnvironment` (always) + per-tab `FeatureContainer` (instantiated on tab open) is the right shape if memory ever becomes a constraint or service count reaches ~50. Until then, the eager-init pattern is right — predictable, view bodies don't nil-check, closure capture works at init time. Don't refactor pre-emptively.
- [ ] **Service init failure recovery (CoreData corruption).** Today if `CoreDataStack.init` fails to load the persistent store — corrupted file, schema-migration failure, disk error, post-iCloud-restore mismatch — the iOS app crashes inside `AppEnvironment.init` before LoginView renders. User has no in-app recovery path; only fix is delete + reinstall, which **destroys every unsynced offline edit**. For a field-worker app that's offline-first, that's the worst possible outcome. **Fix**: wrap `loadPersistentStores` in a recovery path — on failure, log the error (via a crash reporter, see next item), delete the corrupted store file, recreate fresh. User loses local cache for that session but the app launches. Pair with telemetry so we know when it's happening in the wild. Consider an "in-memory CoreData fallback" so writes stay usable for the current session even if persistence is broken. **Add a crash reporter** (Sentry / Firebase Crashlytics / a simple `POST /api/v1/admin/crash-log` endpoint) — today we have no visibility into launch crashes on testers' phones.
- [ ] **Company-switch reset must sync offline edits first.** `AppEnvironment.companyService.onCompanySwitch` (lines 174-198) calls `resetForLogout()` on every service — clears in-memory state but doesn't sync pending edits before context flips. User-initiated logout (`authService.onUserLogout`) correctly uses `prepareForLogout()` which syncs first; company switch should be at least as careful. **Scenario at Phase 5+** when CompanyId scoping is API-enforced: user in Company A clocks in offline → switches to Company B (JWT cid=B) → network returns → `observeConnectivity` fires `syncPendingUpdates` → pending clock-in POSTs with cid=B → API either rejects (edit stuck forever in CoreData) or accepts and writes the row into the wrong tenant (silent corruption). Neither outcome is good. **Fix two layers**: (1) call `prepareForLogout()` not `resetForLogout()` on company switch — same chain `onUserLogout` uses, syncs first if connected; (2) tag every pending CoreData row with a `pendingCompanyId: Int64` set at the moment of the edit, so the sync path can defer pending rows until the right Company is active again. Lands alongside Phase 5 sproc-level CompanyId enforcement on the API. Today: not a bug — only one Company exists. The moment a second Company arrives, becomes a real risk.

## iOS Profile redesign — battle-test follow-ups (2026-05-11)

Items audited and deferred from the pre-ship battle test. Six HIGHs + four MEDIUMs were fixed in the same session; the entries below are the ones we deliberately punted because they're rare, recoverable, or design-call.

- [ ] **M2 — Concurrent token-write race between `changePassword` + `AuthenticatedSession.refresh`.** Both write to `tokenStore.accessToken/refreshToken` without coordination. Realistic conditions: user is mid-modal typing while a background refresh fires AND the responses arrive in inverted order. Worst case = one extra forced re-login. Proper fix: a shared write lock between `AuthService.changePassword` and `AuthenticatedSession.refresh`, OR funnel both through a single token-write coordinator actor. File: `build.one.ios/BuildOne/Services/Auth/AuthService.swift::changePassword`.
- [ ] **M3 — Fast-nav race might surface stale `initialValue` in `TextFieldEditScreen`.** `.onDisappear` fires-and-forgets `Task { await onCommit }`. If the user pops + re-pushes the same field row before the optimistic local write lands on the MainActor queue, the new screen reads stale state. Self-correcting on next refresh. Proper fix: maintain a per-field "pending write" map in the VM so subsequent reads see in-flight values. File: `build.one.ios/BuildOne/Features/Profile/TextFieldEditScreen.swift`.
- [ ] **M5 — `TextFieldEditScreen` has no Cancel.** Chevron-back always commits the typed value (after validation in the caller). Per the Profile spec ("iOS Settings-style: tap row → push → edit inline → back auto-saves"), but iOS convention often pairs that with an explicit Cancel that pops without committing. Design call. If we add Cancel: ToolbarItem(.cancellationAction) that sets `value = initialValue` and then dismisses.
- [ ] **M7 — Theme + biometric-enabled `UserDefaults` keys persist across logout.** `build.one.theme.override` and `build.one.biometric.enabled` are device-scoped. User A picks Dark + enables Face ID → User B logs in fresh and inherits both. Theme is arguably correct as device-preference; biometric-enabled flag is more debatable (the underlying credential is `biometryCurrentSet`-bound so the security boundary is fine — it's the *visual indicator* that mis-states reality for User B's first peek at Profile → Security). Pre-existing pattern, not introduced by the Profile redesign. Fix would be either scoping the keys by userId or clearing them in `resetForLogout()`.
- [ ] **L2 — Verify `.preferredColorScheme` propagation to the iOS status bar** in the simulator across all three Appearance settings. Likely fine — SwiftUI's `.preferredColorScheme` is documented to drive `UIStatusBarStyle` — but worth eyeballing under each Theme override.

## Bill list page review — `build.one.web/src/pages/bills/BillList.tsx` (2026-05-06)

Findings from a focused review of the Bills list page. Listed in priority order; first three are real bugs, rest are UX / hardening.

- [ ] **Vendor / Project names render blank past the dropdown's first page.** [BillList.tsx:134-135](../build.one.web/src/pages/bills/BillList.tsx#L134-L135) calls `useIdNameMap` against `/api/v1/get/vendors` and `/api/v1/get/projects` with no `?page_size=`; [useIdNameMap.ts:14](../build.one.web/src/hooks/useIdNameMap.ts#L14) just `getList`s and builds the map from one page. Fix options: (a) pass `?page_size=10000` (cheap; tenant has ~1100 vendors); (b) add a slim id+name endpoint; (c) join `vendor_name` / `project_name` into the bill list response server-side and drop the maps entirely (best long-term).
- [ ] **Sort is page-local, not global.** [BillList.tsx:146-165](../build.one.web/src/pages/bills/BillList.tsx#L146-L165) sorts `resolvedItems`, which is the current 50-row page only. Clicking a `sortable-th` header looks like a global sort but page 2 sorts independently of page 1 — misleading. Fix: lift sort to the API (`?sort=column&dir=asc|desc` query params on `GET /api/v1/get/bills` → through service → into the paginated sproc's `ORDER BY`). Pairs naturally with #1 if the list response gains `vendor_name` / `project_name`.
- [ ] **Folder-processing poll never cancels on unmount.** [BillList.tsx:92-119](../build.one.web/src/pages/bills/BillList.tsx#L92-L119) uses recursive `setTimeout(poll, 2000)` with no AbortController, no `useRef` cancel handle, and no `useEffect` cleanup. Navigating away during folder processing keeps polling and calls `setState` on an unmounted component (React warns). Fix: store the timer id in a `useRef` + clear in a `useEffect` cleanup, or convert to an `AbortController` and check `signal.aborted` before each setState.
- [ ] **Search input has no debounce.** [BillList.tsx:386-388](../build.one.web/src/pages/bills/BillList.tsx#L386-L388) calls `setSearch` on every keystroke. `usePaginatedList` re-fetches on each change — typing "vendor" fires 6 requests against the prod API. Add a 250–300ms debounce inside `usePaginatedList` (or at the input). Same fix would benefit other list pages that consume the hook.
- [ ] **Disallowed file types are dropped silently in the drop zone.** [BillList.tsx:170](../build.one.web/src/pages/bills/BillList.tsx#L170) filters by `ACCEPTED_TYPES` and skips anything else with no feedback. Drop a `.docx` and nothing happens. Add a per-file error row (`error: "Unsupported type"`) so the user knows why their file disappeared.
- [ ] **Brittle sort comparators on `_date` and `is_draft`.** [BillList.tsx:158-162](../build.one.web/src/pages/bills/BillList.tsx#L158-L162) does `Number(aStr)` first, falls through to `localeCompare`. Move to typed comparators per column (a `getSortValue(col, row)` map returning `number | string`). Lower priority — wait until #2 lifts sort to the server, which makes this moot.
- [ ] **Inconsistent persistence: `sortKey` / `sortDir` use sessionStorage, `statusFilter` does not.** [BillList.tsx:49](../build.one.web/src/pages/bills/BillList.tsx#L49) vs [:50-55](../build.one.web/src/pages/bills/BillList.tsx#L50-L55).
